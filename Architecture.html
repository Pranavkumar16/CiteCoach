CiteCoach V1 Architecture

Offline Document Intelligence • Evidence-Based • Privacy-First

🏗️ Architecture Overview

CiteCoach V1 uses a battle-tested 5-layer offline architecture inherited from DocMind, refined to focus exclusively on per-document RAG chat with citations.

✅ What We Kept from DocMind

  • 5-layer architecture - Proven with 2M+ users
  • Hierarchical retrieval - Page → Chunk (98% memory reduction)
  • 3K token context - Sweet spot for quality/speed/battery
  • Aggressive caching - 50-60% instant responses
  • Device tier adaptation - Auto-adjust to iPhone SE vs 14 Pro
  • Voice-first UX - STT + streaming + TTS patterns
  • Battery policies - Low-power mode reduces context/tokens

❌ What We Removed for V1

  • Bookmarks tab & AI summaries - Not in V1 scope
  • Auto-flashcard generation - Moved to V1.1 (manual save)
  • Cross-document search - Per-document only in V1
  • OCR for scanned PDFs - Digital PDFs only
  • Cloud sync - 100% offline forever
  • Confusion tracker - Removed in DocMind already
💡 V1 Focus

CiteCoach V1 does ONE thing exceptionally well: Evidence-based conversations with your PDFs, completely offline. Every answer includes citations. Tap any citation to jump to the source page. All processing happens on your device.

Naive approach: Load all 10,000 chunks → 40MB RAM → 3.5s → Crashes on iPhone SE

LAYER 4: Context Window Management
Goal: Fit question + context + answer in 3000 tokens without quality loss
Token Budget Allocation (3K Token Sweet Spot)
Total context: 3000 tokens (proven optimal)

Breakdown:
• System Prompt: 200 tokens (7%)
  - Role definition
  - Citation format instructions
  - "Always cite page numbers"

• Question: 50 tokens (2%)
  - User's actual question

• Retrieved Context: 2,500 tokens (83%)
  - 5-8 chunks × 300-500 tokens each
  - Page markers for citation tracking
  - Overlap for continuity

• Response Buffer: 250 tokens (8%)
  - Space for LLM answer + citations
Citation Tracking System
How citations work:
1. Each chunk knows its source page number
2. Context includes page markers: "[Page 67] text..."
3. System prompt instructs: "Always cite pages like (p.67)"
4. LLM generates: "...mitochondria produce ATP (p.67, p.68)..."
5. UI parses citations and makes them tappable
6. Tap → Navigate to PDF page 67
Adaptive Context by Question Type
Question Type Context Budget Chunks Used

📄 Document Import & Processing Pipeline
1
PDF Import (Local File Picker Only)
• iOS Files app / Android file picker
• No cloud storage options in V1
• 100% offline principle maintained
• File copied to app's documents directory
2
Text Extraction (PDF.js / PDFKit)
• Extract text page by page
• Digital PDFs only (no OCR in V1)
• Store in SQLite: pages table
• Typical 127-page textbook: ~2-5 seconds
3
Smart Chunking (Semantic Boundaries)
DocMind proven strategy:
• Split at headers, paragraphs, sentences
• Variable size: 200-600 tokens
• 50-token overlap between chunks
• Store chunk_text + page_number + start_offset
• Typical 127-page doc: ~800-1200 chunks
4
Embedding Generation (TinyBERT 25MB)
• Convert each chunk to 384-dim vector
• Process: 30-50 chunks/second on device
• Store embeddings in SQLite
• Typical doc: 20-40 seconds total
• Storage: 800 chunks × 1.5KB = 1.2MB embeddings
5
Page-Level Index Creation
Hierarchical retrieval setup:
• Aggregate chunk embeddings per page
• Create page-level embedding (average of chunks)
• Store in pages table
• Enables fast Stage 1 retrieval
• Memory footprint: 127 pages × 1.5KB = 190KB
6
Processing Complete → Ready
• Document status: READY
• User can now start chatting
• All embeddings pre-computed (never recompute)
• Hierarchical index built
• NO auto-flashcards in V1 (removed)

🎯 Why This Works on Low-End Devices

Hierarchical retrieval is the key:
• We NEVER load all 10,000 chunk embeddings (40MB)
• We load 127 page embeddings (190KB) → find top 5 pages
• We load 50 chunk embeddings (100KB) → find top 8 chunks
• Peak memory for retrieval: 500KB vs 40MB naive

This is why CiteCoach works on iPhone SE while naive approaches crash.

🛡️ How This Architecture Survives Real Constraints

Constraint 1: Low RAM (4GB iPhone SE)

Problem: Naive approach loads 40MB of embeddings → crash

Solution: Hierarchical retrieval (page → chunk) uses <500KB peak

Result: Works on all devices, no crashes

Constraint 2: Slow Token Generation (1-2 tok/s)

Problem: 250-token answer = 2 minutes wait → user quits

Solution: Streaming + device tier adaptation (120 tokens on Tier C)

Result: Perceived 10s wait, user sees progress

Constraint 3: Battery Drain (2% per answer)

Problem: 15 questions = 30% battery → app deleted

Solution: Caching (50% hit) + low-power mode (1800 token context)

Result: 8-12% battery per hour (acceptable)

Constraint 4: Storage Limits

Problem: Embeddings + cache → 100s of MB per doc

Solution: Efficient storage (1-2MB embeddings, 5MB cache)

Result: ~10-25MB per document total
